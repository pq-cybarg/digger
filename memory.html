Memory forensics Live VM region inspection, anomaly detection, optional YARA-on-memory. Cross-platform: Linux /proc, macOS vmmap, Windows ctypes.

What it catches

Snapshot-mode memory forensics can't do Volatility-grade reconstruction (that needs a full dumped image and a profile), but it gets you the three highest-signal indicators visible from VM region metadata alone:

IndicatorWhat it meansSeverity
RWX regions A page is simultaneously readable, writable, and executable. Some JIT runtimes do this legitimately (V8, .NET, Lua, Java HotSpot tier-up) but a plain native process holding RWX pages is a strong process-injection tell. high
Anonymous executable regions An executable VM region with no on-disk file backing. Shellcode landing pads look exactly like this — bytes mapped via mmap(PROT_EXEC) with no file behind them. medium
Library loaded from a drop location A module mapped from /tmp, /Users/Shared, %TEMP%, etc. Sideloaded malicious dylibs/DLLs appear here. high

The memory_anomaly detector fires one finding per category per process, attached to the memory_regions artifact from that process. The collector emits at most 200 suspect regions per process; the full region map can run into tens of thousands of pages on something like Chrome.

Standard pipeline

Memory collection runs as part of digger collect by default — the new memory_regions collector iterates every PID the user can read and records suspect regions. The anomaly detector then runs as part of digger scan.

digger collect --case-dir ./case-1
digger scan    --case-dir ./case-1
# memory_anomaly findings appear alongside everything else

To skip memory collection (it's the slowest collector on a busy host):

digger collect --case-dir ./case-1 --only system,env,processes,network

Ad-hoc CLI

# Scan every readable process and report suspect regions
digger memory scan

# Limit to one PID and add YARA on the dumped bytes
digger memory scan --pid 4242 --yara

# Dump every suspect region of one PID to disk for offline analysis
digger memory dump --pid 4242 --out-dir ./dumps

# Dump a specific region by start address
digger memory dump --pid 4242 --addr 0x7f1234500000 --out-dir ./dumps

Output of memory scan:

pid=4242 evil — 2 suspect region(s) of 87 total

YARA on memory

When --yara is set, each dumped region is matched against the bundled YARA rules (digger/rules/yara/) plus the signature-base YARA corpus if you've run digger loki update. Matches print under each region:

  rwxp 0x… — [ANON-EXEC] [anon]
     YARA: rule=digger_mimikatz_strings tags=['credential']
     YARA: rule=APT_Cobalt_Strike_Beacon tags=['actor', 'apt']

YARA on memory finds things YARA-on-disk can't: unpacked payloads, in-memory keys, deobfuscated strings.

Permissions & platform support

Region infoByte dumps
Linux /proc/[pid]/maps — own user can read own processes; root reads all /proc/[pid]/mem — needs CAP_SYS_PTRACE (root or yama ptrace_scope ≤ 0)
macOS vmmap — works for own user Not supported (would require lldb attachment). Region anomaly detection only.
Windows OpenProcess + VirtualQueryEx via ctypes — best-effort ReadProcessMemory — needs SeDebugPrivilege (admin)

The collector gracefully degrades: processes whose maps can't be read are silently skipped (logged at info level). It never raises.

What's NOT included

  • Full memory imaging. No winpmem / LiME- style raw RAM capture. The bytes you get are per-region read-only snapshots, not a single coherent image.
  • Pool tag scanning / kernel object reconstruction. Reserved for proper Volatility / Rekall workflows.
  • Page-protection history. We see current permissions, not what they were when the page was first allocated — so a region that used to be RW and was then mprotect()'d to RX (a classic injection pattern) looks identical to a region that was always RX. Live monitoring would catch the transition; snapshot cannot.
  • Per-thread stacks. Anonymous executable thread stacks are surfaced; per-thread CPU state isn't.

Programmatic API

from digger.memory import (
    list_regions_for_pid, list_regions_for_all_pids,
    dump_region, yara_scan_region, can_dump_pid,
)

# Snapshot one process
regions = list_regions_for_pid(4242)
for r in regions:
    if r.is_rwx or r.is_anonymous_exec:
        print(r.perms, hex(r.start), r.size, r.backing)

# Best-effort byte dump (works on Linux as root, Windows with SeDebugPrivilege)
ok, reason = can_dump_pid(4242)
if ok:
    for r in regions:
        if r.is_anonymous_exec:
            data = dump_region(r)
            # ... your own analysis here ...
            hits = yara_scan_region(r)
            for h in hits:
                print("yara:", h["rule"], h["tags"])
