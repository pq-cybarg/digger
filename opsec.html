Operator opsec The rest of digger protects the evidence. This module protects the investigator: bundle encryption, redaction, counter-surveillance, air-gap mode, secure wipe, self-recognition.

Why this exists

Forensic investigators sometimes work on hosts where they themselves could be the target — live-response on a compromised endpoint, an engagement against an adversary with anti-forensics tooling, or just day-to-day handling of cases that contain sensitive identifying information. The opsec module covers:

Confidentiality at rest
PQC-KEM + AES-256-GCM encrypted case bundles. One signed file, decryptable only by the intended recipient.
Confidentiality on share
PII redaction — pseudonymize usernames / hostnames / private IPs / paths consistently across a whole case, with a reversible mapping kept separately.
Detecting observation
Counter-surveillance — enumerate processes that may be watching the investigation: ptrace-attached debuggers, packet captures, EDR/AV agents, screen recorders, eBPF programs, TCC-permitted apps.
Network discipline
Air-gap mode — refuse every outbound HTTP path (intel feeds, LLM triage, TAXII push) with an audit-loggable AirgapViolation.
Disposal
Secure wipe — multi-pass overwrite then unlink of a case directory; refuses to wipe paths that don't look like digger cases.
Self-awareness
Self-identification — every hunt and detector that might catch digger's own activity annotates the row with "this is digger" instead of producing a confusing alert.

The commands

digger opsec status                       # JSON dump of operator posture
digger opsec watchers [-v]                # enumerate observers

digger opsec encrypt --case-dir DIR --out BUNDLE.digger
       --recipient PATH-TO-RECIPIENT-PUBKEY
       [--kem-alg ML-KEM-768]
       [--sign-key PATH-TO-SECRET-KEY [--sig-alg ML-DSA-65]]

digger opsec decrypt --in BUNDLE.digger --key PATH-TO-SECRET-KEY
       --out-dir RESTORE-PATH [--no-verify-sig]

digger opsec redact --case-dir DIR --out-dir REDACTED
       [--keep-hostnames] [--keep-usernames] [--keep-raw-blobs]
       [--redact-public-ips]

digger opsec wipe --case-dir DIR --yes [--passes 3]

Global flag:

digger --airgap <any-command>            # enter air-gap mode for this invocation
DIGGER_AIRGAP=1 digger ...                # same, via env

Encrypted bundles

digger opsec encrypt compresses a whole case directory (tar.gz), wraps it with an AES-256-GCM key derived from a PQC-KEM- encapsulated shared secret, and writes a single .digger file. The header records the KEM algorithm, the recipient's public-key SHA-256 fingerprint, the case ID and the dual chain-tip hashes — so the recipient can verify the bundle binds to the chain state they expect.

Optionally signs the entire header+ciphertext with a PQC signature (default ML-DSA-65, FIPS 204). Self-contained — the signing public key travels with the bundle, so verification is a single operation.

# Generate a recipient ML-KEM-768 keypair (one-time)
python -c "from digger.crypto import PQCBackend; b=PQCBackend(kem_alg='ML-KEM-768'); pk,sk=b.generate_kem_key(); open('recipient.pk','wb').write(pk); open('recipient.sk','wb').write(sk)"

# Encrypt + sign (signing key was previously created by `digger pqc sign`)
digger opsec encrypt --case-dir ./case-2026-05-21 \
       --out case-2026-05-21.digger \
       --recipient recipient.pk \
       --sign-key /secrets/case-2026-05-21.sk

# Decrypt + verify on the receiving end
digger opsec decrypt --in case-2026-05-21.digger \
       --key recipient.sk \
       --out-dir ./restored/

FIPS mode (--fips-mode) restricts the choices to ML-KEM + ML-DSA / SLH-DSA / Falcon.

Redaction

For sharing a case with a vendor, ISAC, or partner CSIRT. The redactor walks every artifact and finding, applying consistent pseudonyms — same username always maps to the same USER001, private IPs to PRIV-IP1, hostnames to HOST1. The reversible map is written to redaction_map.json in the output directory; the case custodian keeps it separately to enable de-redaction if needed.

Default policy (override with flags):

WhatDefault
Emailsredacted
MAC addressesredacted
Private IPs (RFC 1918, loopback excluded)redacted
Public IPskept (often intel-relevant)
Usernames in home-dir pathsredacted
Hostnames in case metaredacted
AWS 12-digit account IDsredacted
Raw blob fields (raw, contents, tail, lines)dropped
Environment-variable valuesdropped (keys kept)

The dual chain hash is regenerated from the redacted content so the output passes digger verify as a valid (different) case.

Counter-surveillance

digger opsec watchers enumerates processes whose presence implies someone is observing the host or the investigation:

CategoryWhat it catchesSeverity
debuggergdb, lldb, dtrace, strace, ltrace, rr, plus an explicit ptrace check on Linux (TracerPid on our own /proc/self/status)high
packet_capturetcpdump, tshark, wireshark, dumpcap, ngrep — every byte digger sends is being recordedhigh
edrCrowdStrike, SentinelOne, Carbon Black, Defender ATP, Sysmon, osquery, Velociraptor, ESET, Sophos, …medium
recorderOBS, Loom, ScreenFlow, Snagit, Camtasia, QuickTime screen capturemedium
ebpfActive eBPF programs attached (via bpftool prog list)medium
auditauditd, splunkd, splunk-forwarder, rsyslog, audit/log shipperslow
tccmacOS apps with kTCCServiceScreenCapture permissionlow

Discovery only — no process is touched. The presence of a watcher doesn't mean digger is being targeted; it means the analyst should know what's running and adjust opsec accordingly (kill the screen recorder, disable EDR's data forwarding, switch to a write-blocked image, etc.).

Air-gap mode

Every network-egress path in digger is gated by assert_network_allowed(). With --airgap on, attempts raise AirgapViolation and increment a counter the status command surfaces:

$ DIGGER_AIRGAP=1 digger intel update --only cisa_kev
digger.opsec.airgap.AirgapViolation: air-gap mode is enabled;
refusing network operation: intel-feed:cisa_kev

$ digger opsec status
{
  "airgap": {
    "enabled": true,
    "attempted_violations": 1,
    "last_violation": "intel-feed:cisa_kev"
  }, ...
}

Gated call-sites: every intel feed fetcher, the LLM chat client, every TAXII GET/POST. The opsec module itself is fully local.

Self-identification

When a hunt or detector catches a row that's actually digger doing its job (e.g. interpreter-in-temp catching python invoking digger collect --case-dir /tmp/...), the row is rendered with a green diamond marker and an explicit self column ("digger via python [collect]") instead of silently filtering. The signal isn't lost — it's just clearly labeled.

Available to your own hunts:

from digger.opsec.self_id import identify
attribution = identify(process_data_dict)
# → "digger via python [scan]"  or  None

Secure wipe

Multi-pass random overwrite then unlink. Refuses any path that doesn't contain a digger sentinel file (evidence.db or chain_of_custody.json) as a safety check — you can't accidentally wipe ~/.

SSD caveat. On modern flash storage with wear-leveling controllers, block-level overwrite is best-effort. Adversaries with physical access to the disk can recover overwritten blocks via the controller's spare-block pool. For real confidentiality at rest on SSDs, rely on full-disk encryption rather than overwrite.

Status

digger opsec status returns a JSON snapshot suitable for piping into jq or aggregating in a fleet view. Includes: FIPS posture, air-gap state + violation counter, traffic summary, digger's own PIDs (per self_id), watcher rollup by category + full details of high-severity watchers.

Putting it together — a paranoid run

digger --fips-mode --airgap \
       --classification "Confidential" \
       --tlp TLP:AMBER+STRICT \
       investigate --case-dir ./case-2026-05-21

digger opsec watchers           # check who saw the run
digger opsec encrypt --case-dir ./case-2026-05-21 \
       --out case-2026-05-21.digger \
       --recipient /secrets/peer.pk \
       --sign-key /secrets/operator.sk
digger opsec wipe --case-dir ./case-2026-05-21 --yes
# bundle is the only artifact left
